# AGENTS.md

## 通用说明

- 当对代码、配置、接口、运行方式或用户可见行为做出修改时，需要同步更新对应的文档，确保文档与实际实现保持一致。
- 修改功能后如果会影响用户使用方式、启动步骤、配置项、接口调用或验证流程，需要同步更新 `README.md`，确保快速入门文档与实际行为一致。
- 每次修改功能前都需要先查看 `docs/` 下的相关文档，理解当前功能状态、协议约束和系统约定后再开始改动。
- 开发时需要为关键逻辑、复杂流程和不易理解的约定补充必要注释，注释统一使用中文。
- 所有方法、属性、常量、枚举值、record 组件、配置 Bean 和公开扩展点都需要补充中文注释；简单 getter/setter 可通过字段注释和类级说明覆盖，但不能让读者需要猜测字段含义、协议含义或生命周期边界。
- 所有类创建的时候加上作者 @author kongweiguang
- 测试要尽量完整、覆盖全面，优先覆盖协议解析、异常输入、状态机边界、异步 `turnId` 隔离、打断流程、配置默认值和公开扩展点；新增功能必须同步补充能保护长期行为的测试。
- 测试需要做好分类，按领域使用清晰的包名、类名、`@Tag` 和 `@DisplayName` 标注，例如 `protocol`、`pipeline`、`audio`、`session`、`turn`，方便 IDE、Maven 和 CI 按类别查看与筛选。
- 当前项目会作为 GitHub 开源项目维护，代码、配置、文档和示例都要按公开仓库标准处理，避免提交本地 IDE、临时调试、机器私有路径、明文密钥和只适合个人环境的默认值。
- 方案选择优先考虑中长期可维护性、健壮性和扩展性，不为了短期通过验证牺牲清晰分层、协议兼容、错误边界、测试覆盖和后续真实服务替换空间。
- 设计开发时优先采用成熟设计模式表达稳定变化点，例如 Strategy、Factory、Observer/Hook、Facade、State、Adapter、Template Method；使用模式必须服务于解耦、替换、测试或扩展，不为了形式引入过度抽象。
- 已存在的 ASR/VAD/LLM/TTS 可替换接口、每会话 ASR 工厂、流水线 hook、TurnManager 状态机和 WebSocket 适配层应作为后续设计的参考，不要绕过这些模式直接把业务逻辑写进核心编排类。
- 修改代码时优先复用现有分层与扩展点；如果必须新增抽象，需要说明它解决的长期复杂度，避免为单个临时场景引入过度设计。
- 对外协议、公开接口、配置项和 mock 行为要尽量稳定、类型清晰，并通过文档和测试固化边界，避免让调用方依赖隐式约定。

## 文档索引

- `README.md`：面向新手的快速入门指南，用于帮助初次接触项目的人快速理解、启动和验证系统。
- `docs/`：面向开发过程使用的文档集合，用于记录功能状态、协议约束、系统约定和长期开发上下文。
- `docs/features.md`：记录项目功能清单、实现状态、验收点和后续关注事项。
- `docs/frontend-integration.md`：面向 Web 前端联调，说明连接后端、发送文本或音频、处理下行事件和使用 demo 验证的方法。
- `docs/protocol.md`：定义 WebSocket 上下行消息协议、示例和交互约束。
- `docs/system-state.md`：保存跨多轮开发的当前目标、硬约束、已完成内容、待决问题和下一步计划。
